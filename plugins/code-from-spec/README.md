# code-from-spec

Spec 문서를 기반으로 코드를 구현하고, **Spec 준수도 점수**를 산출하며, 코드 리뷰와 Spec 피드백까지 자동화하는 플러그인.

## 에이전트 구조

```
impl-orchestrator (조율자)
├── work-scheduler      Spec 의존성 분석 → Jira 티켓 → Git 브랜치
├── code-generator      Spec → 코드 구현 (Agent Teams 4명)
├── spec-verifier       코드 → Spec 준수도 검증 (100점)
└── spec-feedback       모호점 + 갭 + PR 리뷰 → Spec 수정 제안
```

## Full 파이프라인

```
  Spec 문서들
      │
      ▼

 │     Git 브랜치 생성 (Spec 1개 = 브랜치 1개)           │
```

## 워크플로우

| 워크플로우 | 트리거 프롬프트 | 하는 일 |
|-----------|----------------|---------|
| `full` | "전체 파이프라인 실행해줘" | 위 전부 한 번에 |
| `schedule` | "작업 계획 세워줘" | 의존성 분석 + Jira + 브랜치 |
| `implement` | "구현해줘" | Spec → 코드 생성 |
| `code-review` | "코드 리뷰해줘" | 체크리스트 리뷰 → 수정 루프 |
| `verify` | "검증해줘" | Spec 준수도 100점 평가 |
| `feedback` | "Spec 보강해줘" | 모호점+갭 → Spec 수정 제안 |
| `review-apply` | "PR 리뷰 반영해줘" | 코드 수정 + 리뷰 루프 + 재검증 + Spec 동기화 |

## 실행 모드

| 모드 | 설명 | 사용 시점 |
|------|------|---------|
| `auto` | Level 0→N 연속 구현 | 신뢰도 높은 Spec |
| `review-gate` | Level마다 PR → 사람 리뷰 → 승인 후 다음 | 초기/중요 도메인 |

```
review-gate 모드:

  Level 0 구현 → PR 생성 → ⏸ 리뷰 대기
                                │
                    approved ←──┤──→ 수정 피드백
                       │                  │
                       ▼                  ▼
                 Level 1 구현       코드 수정 → PR 업데이트 → ⏸
                       │
                      ...반복
```

## 코드 리뷰

프로젝트 유형(Spring/React)을 자동 감지하여 체크리스트를 로드합니다.

### 리뷰 기준 3소스 병합

```
소스 A: code-review 스킬 (활성화 시) → 범용 리뷰 프레임워크
소스 B: review-spring.md / react.md  → 기술 스택 체크리스트
소스 C: 프로젝트 CLAUDE.md / yml     → 프로젝트 전용 규칙
         ↓
       합집합 병합 (충돌 시 소스 C 우선)
```

### 우선순위 분류

| 레벨 | 대상 | 조치 |
|------|------|------|
| **P1 Critical** | 하위 호환성, 보안, 데이터 안정성 | 반드시 수정 |
| **P2 Warning** | 검증 책임, 레이어 분리, JPA, 성능, 테스트 | 수정 권장 |
| **P3 Suggestion** | 코드 스타일, 가독성 | 선택적 |

### Spec 갭 정책

코드 리뷰에서 P1/P2 이슈가 발견되었는데 Spec에 해당 항목이 없는 경우:
- "Spec에 명시되지 않았으므로 준수"라는 판단을 **금지**
- Spec이 불완전한 것으로 간주 → 코드 수정 + Spec 보강

## 구현 추적

Spec 파일에 구현 상태를 기록합니다 (Spec = SSOT):

```yaml
## 구현 추적
status: not-started | in-progress | completed | review-pending | review-approved | verified
jiraTicket: PROJ-123
branch: PROJ-123-spec-id-kebab
pr: https://github.com/...
commit: abc123def...
completedAt: 2026-03-25T10:30:00Z
```

이미 `completed`/`verified` 상태인 Spec은 스케줄링에서 자동 제외됩니다.

## 사용 예시

```
"이 Spec들의 작업 계획을 세워줘"                   # schedule
"이 Spec으로 프로젝트를 구현해줘"                  # implement (auto)
"리뷰 포함해서 구현해줘"                           # implement (review-gate)
"이 코드 리뷰해줘"                                 # code-review
"구현이 Spec을 잘 따르는지 검증해줘"               # verify
"이 PR 리뷰 반영해줘 {PR_URL}"                     # review-apply
"Spec 기반으로 구현하고 준수도도 확인해줘"          # full
```
