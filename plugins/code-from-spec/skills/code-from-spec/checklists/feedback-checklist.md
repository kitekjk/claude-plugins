# 피드백 완료 체크리스트

## 사전 조건

### F-00: 피드백 전 확인
- [ ] 구현 디렉토리에 `ambiguity-log.md` 존재
- [ ] 검증 리포트(`reports/compliance-{date}.md`) 존재
- [ ] 원본 Spec 디렉토리 존재

## 소스 수집

### F-01: 모호점 로그 분석
- [ ] ambiguity-log.md의 모든 AMB 항목 추출
- [ ] 각 항목의 Spec 파일/섹션 매핑 확인
- [ ] 중복 항목 통합

### F-02: 검증 Spec 갭 분석
- [ ] 검증 리포트에서 원인이 "Spec갭"인 항목 추출
- [ ] 각 항목의 Spec 파일/섹션 매핑 확인
- [ ] 모호점 로그와의 중복 확인

### F-02-1: PR 리뷰 피드백 분석 (PR 리뷰 소스가 있는 경우)
- [ ] PR 리뷰 코멘트 또는 리뷰 노트에서 코드 변경 항목 추출
- [ ] 각 변경 항목을 대응하는 Spec 파일/섹션에 매핑
- [ ] 코드 변경이 Spec과 달라진 부분만 REV-{nn}으로 기록
- [ ] AMB/Spec갭 항목과의 중복 확인

## 수정 제안

### F-03: 카테고리 분류
- [ ] 각 항목을 6개 카테고리 중 하나로 분류
- [ ] 카테고리별 분포 확인

### F-04: 수정 제안 생성
- [ ] 각 항목에 대해 구체적 수정 내용 작성
- [ ] 현재 Spec 내용과 수정 제안의 차이 명시
- [ ] 수정 근거 (구현 판단 내용) 포함

## 승인 및 반영

### F-05: 사용자 승인
- [ ] 수정 제안 목록을 사용자에게 제시
- [ ] 승인/거부/수정 결과 기록

### F-06: Spec 반영
- [ ] 승인된 항목만 Spec 파일에 반영
- [ ] 기존 내용의 의미가 변경되지 않았는지 확인
- [ ] 새 규칙과 기존 규칙 간 충돌 없는지 확인

## 리포트

### F-07: 리포트 완결성
- [ ] 카테고리별 분포 포함
- [ ] 모든 항목의 승인 상태 기록
- [ ] 수정된 Spec 파일 목록 포함
- [ ] `reports/feedback-{date}.md`에 저장
