# 코드 리뷰 체크리스트 — Spring (Java/Kotlin)

프로젝트 유형이 Spring일 때 코드 리뷰 루프에서 사용하는 **범용** 체크리스트.
프로젝트 전용 규칙은 런타임에 아래 소스에서 동적으로 병합한다:

| 소스 | 역할 |
|------|------|
| `code-review` 스킬 (활성화 시) | 범용 리뷰 프레임워크 (Pre-Comment Checklist, 중복 방지 등) |
| 이 파일 (`review-spring.md`) | Spring 기술 체크리스트 |
| 프로젝트 CLAUDE.md / AGENTS.md | 프로젝트 아키텍처·컨벤션 규칙 |
| `.github/workflows/claude-review.yml` | CI 리뷰 봇과 동일한 프로젝트 전용 규칙 |

---

## 우선순위 분류

리뷰 이슈 발견 시 아래 우선순위로 분류한다. P1은 반드시 수정, P2는 수정 권장, P3는 선택적 개선.

| 우선순위 | 대상 | 조치 |
|---------|------|------|
| **P1 Critical** | 하위 호환성 파괴, 데이터 손실, 보안 취약점, SPI 계약 위반 | 반드시 수정 |
| **P2 Warning** | 성능 문제, 테스트 누락, 예외 처리 부족, 캐시 무효화 누락 | 수정 권장 |
| **P3 Suggestion** | 코드 스타일, 네이밍, 문서화, 미세 개선 | 선택적 개선 |

## 수정 제안 제한 원칙

- 변경된 코드에 대해서만 리뷰한다 (변경하지 않은 코드에 대한 "정렬" 목적 수정 금지)
- 명확한 문제가 있을 때만 제안한다 (취향/선호도 제외)
- 실질적 가치가 있는 개선만 지적한다 (코드 동작에 영향 없는 스타일 제외)
- 프로젝트 기존 패턴과 명확히 다를 때만 컨벤션을 지적한다
- 관련 없는 API/메서드를 비교하여 일관성을 요구하지 않는다
- 일반적이거나 뻔한 개선은 건너뛴다 (변수명 약간 더 명확하게, 주석 추가 등)

## Pre-Comment Checklist

이슈를 리포트하기 전에 반드시 확인:

1. **Not Duplicate** — 이전에 이미 지적하고 해결된 이슈가 아닌가
2. **In Scope** — 이번에 실제로 변경된 코드인가
3. **Independent** — 관련 없는 API/메서드와 비교하고 있지 않은가
4. **Meaningful** — 실질적 영향이 있는 이슈인가

---

## P1 Critical — 반드시 수정

### R-01: 하위 호환성 / 계약

- [ ] 기존 API 시그니처(파라미터, 응답 구조)가 변경되었다면 하위 호환이 유지되는가
- [ ] Enum 이름/값 변경이 기존 저장 데이터를 깨뜨리지 않는가
- [ ] 외부 시스템 연동 인터페이스의 계약이 유지되는가

### R-02: 보안

- [ ] API 키/시크릿이 하드코딩되어 있지 않은가
- [ ] 로그에 민감정보(토큰, 개인정보)가 노출되지 않는가
- [ ] 외부 시스템 credential이 환경변수/시크릿 관리되는가
- [ ] SQL 인젝션 취약점이 없는가 (네이티브 쿼리 사용 시)
- [ ] 인증/인가 체크가 누락된 엔드포인트가 없는가
- [ ] 입력값 검증(@Valid, @NotNull 등)이 누락되지 않았는가

### R-03: 데이터 안정성

- [ ] null 참조 가능성이 적절히 처리되었는가 (Optional, null 체크)
- [ ] 경계값(빈 리스트, 0, 음수, 최대값)이 처리되었는가
- [ ] 트랜잭션 범위가 올바른가 (부분 커밋/롤백 위험 없는가)

---

## P2 Warning — 수정 권장

### R-04: 검증 책임 (도메인 우선)

- [ ] 비즈니스 검증 로직이 도메인 모듈(엔티티, VO, Enum, 도메인 서비스)에 있는가
- [ ] API 레이어(@Valid, @NotBlank 등)에만 검증이 있고 도메인에 검증이 없지 않은가
- [ ] 도메인 검증 위반 시 명확한 에러 클래스/에러 코드를 반환하는가
- [ ] 같은 검증 규칙이 여러 레이어에서 중복 구현되었다면, 도메인이 원본이고 API는 fast-fail 보조인가

### R-05: 레이어 분리 / 의존성 방향

- [ ] 의존성이 단방향인가 (상위 레이어 → 하위 레이어)
- [ ] 도메인/코어 모듈이 인프라/프레젠테이션 모듈을 참조하지 않는가
- [ ] @Transactional이 적절한 레벨(Service/UseCase)에만 있는가
- [ ] 엔티티/VO를 API에 직접 노출하지 않고 DTO로 변환하는가

### R-06: JPA / 데이터 접근

- [ ] N+1 문제가 없는가 (FetchJoin/EntityGraph/BatchSize 사용 확인)
- [ ] 읽기 전용 조회에 불필요한 영속성 컨텍스트 로딩이 없는가
- [ ] 벌크 연산 시 영속성 컨텍스트 초기화를 수행하는가
- [ ] 인덱스가 필요한 쿼리에 적절한 인덱스가 있는가

### R-07: 성능

- [ ] 불필요한 DB 조회가 반복되지 않는가
- [ ] 캐시 적용이 필요한 데이터 조회에 캐시가 있는가
- [ ] 데이터 수정 시 관련 캐시 무효화가 빠지지 않았는가
- [ ] 대량 데이터 처리 시 배치/페이징을 사용하는가
- [ ] 비효율적 알고리즘이나 불필요한 반복 처리가 없는가

### R-08: 에러 핸들링

- [ ] 도메인별 커스텀 예외를 프로젝트 컨벤션에 맞게 사용하는가
- [ ] 예외 메시지가 디버깅에 충분한 정보를 포함하는가
- [ ] checked exception을 무시하거나 빈 catch 블록이 없는가
- [ ] 외부 시스템 호출에 적절한 fallback/retry가 있는가

### R-09: 테스트

- [ ] 새로운 로직에 대응하는 테스트가 있는가
- [ ] 테스트 구조가 프로젝트 컨벤션을 따르는가
- [ ] Mock 사용이 과도하지 않은가 (통합 테스트 우선)
- [ ] 경계값/예외 케이스 테스트가 있는가

### R-10: 비동기 / 이벤트 (해당 시)

- [ ] 메시지 Consumer에 멱등성 처리가 있는가
- [ ] 재시도 대상 예외가 올바르게 구분되는가
- [ ] Dead Letter Queue 핸들링이 있는가
- [ ] 이벤트 발행과 트랜잭션 경계가 올바른가

---

## P3 Suggestion — 선택적 개선

### R-11: 코드 품질

- [ ] 불변 객체 패턴을 적절히 사용하는가 (record, final 등)
- [ ] 미사용 import/변수가 없는가
- [ ] 프로젝트 포맷터(Spotless, Checkstyle 등) 규칙을 따르는가
- [ ] 불필요한 복잡도나 중복 코드가 없는가
- [ ] 가독성과 유지보수성이 적절한가
