# 피드백 절차 스펙

## 목적

구현/검증 과정에서 발견된 Spec 모호점과 갭을 수집하여 Spec 수정 제안을 생성하는 절차를 정의한다.

## 사전 조건

- 구현 디렉토리에 `ambiguity-log.md`가 존재 (code-generator가 생성)
- 검증 리포트(`reports/compliance-{date}.md`)가 존재 (spec-verifier가 생성)
- 원본 Spec 디렉토리가 존재

## 절차

### Phase 1: 피드백 소스 수집

두 소스에서 피드백 항목을 수집한다.

| 소스 | 파일 | 추출 대상 |
|------|------|-----------|
| 모호점 로그 | `{impl_dir}/ambiguity-log.md` | AMB-{nn} 항목 전체 |
| 검증 리포트 | `reports/compliance-{date}.md` | 원인이 "Spec갭"인 미준수 항목 |

### Phase 2: 카테고리 분류

각 항목을 보강 대상 Spec 카테고리로 분류한다.

| 카테고리 | 보강 대상 | 수집 기준 |
|----------|-----------|-----------|
| 비즈니스 규칙 모호 | policies/ | 판정/계산 규칙이 불명확 |
| Spec 흐름/조건 미명시 | usecase/service Spec | 응답 필드/상태 코드/흐름 단계 미정의 |
| 도메인 모델 모호 | service-definition / model Spec | 엔티티 관계/값 범위 불명확 |
| 아키텍처 규칙 부족 | architecture-rules | 레이어 간 규칙 미명시 |
| 네이밍 규칙 부족 | naming-guide | 특정 패턴의 네이밍 미정의 |
| 테스트 시나리오 부족 | Spec 테스트 시나리오 섹션 | 경계값/예외 시나리오 누락 |

### Phase 3: 수정 제안 생성

각 항목에 대해 구체적인 수정 내용을 작성한다.
- Spec 파일 경로 + 섹션
- 현재 내용 (또는 "미기술")
- 수정/추가할 내용
- 수정 근거

### Phase 4: 사용자 승인

- 수정 제안 목록을 사용자에게 제시
- 전체 승인 / 개별 승인 / 수정 후 승인 가능
- 승인된 항목만 반영

### Phase 5: Spec 반영

승인된 항목을 Spec 파일에 반영한다.

반영 규칙:
- 기존 내용의 의미를 변경하지 않음
- 모호한 부분을 구체화하는 방식으로 추가
- 새 규칙 추가 시 기존 규칙과 충돌 여부 확인

### Phase 6: 리포트 저장

`reports/feedback-{date}.md`에 피드백 결과를 저장한다.
