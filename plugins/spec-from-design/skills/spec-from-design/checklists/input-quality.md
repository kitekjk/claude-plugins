# 입력 품질 체크리스트 (input-quality)

> **사용 시점**: design-analyzer 진입 전 orchestrator가 실행
> **목적**: HLD/LLD 입력 문서가 Spec 생성에 충분한 정보를 포함하는지 검증

---

## 사용 방법

각 항목을 확인하고 `[ ]` → `[x]` 로 표시합니다.
`blockOnMissing: true`인 항목이 누락되면 orchestrator가 사용자에게 보고하고 진행을 중단합니다.
사용자가 명시적으로 오버라이드(`--skip-input-check`)를 지정하면 경고만 남기고 진행할 수 있습니다.

---

## HLD 필수 항목

> HLD는 선택 입력이지만, 제공된 경우 아래 항목이 반드시 존재해야 합니다.

| ID | 항목 | blockOnMissing | 설명 |
|----|------|----------------|------|
| H-01 | 서비스 개요 / 스코프 | false | 서비스가 무엇을 하는지, 어느 범위를 다루는지 |
| H-02 | 용어 정의 | false | 도메인 용어, 약어, 핵심 개념 정의 |
| H-03 | KDD (핵심 설계 판단) | true | 아키텍처 결정, 기술 선택 근거, 트레이드오프 기록 |
| H-04 | NFR (비기능 요구사항) | false | 성능, 가용성, 보안, 확장성 등 수치 포함 |
| H-05 | 상태 전이 모델 | false | 도메인 엔티티의 상태 및 전이 조건 |
| H-06 | System Context | false | 외부 시스템 연동, 서비스 경계 다이어그램 |
| H-07 | 이해관계자 / 제약조건 | false | 팀, 기한, 기술 제약, 규제 요건 |

### HLD 체크 항목

- [ ] **H-01** 서비스 개요 및 스코프가 기술되어 있는가?
- [ ] **H-02** 도메인 용어 및 약어 정의가 존재하는가?
- [ ] **H-03** KDD(핵심 설계 판단)가 최소 1개 이상 존재하는가? _(blockOnMissing)_
- [ ] **H-04** NFR이 수치 기반으로 기술되어 있는가? (예: 응답시간 < 200ms)
- [ ] **H-05** 핵심 도메인 엔티티의 상태 전이 모델이 존재하는가?
- [ ] **H-06** 외부 시스템 연동 목록 또는 System Context가 존재하는가?
- [ ] **H-07** 이해관계자, 팀, 기술 제약조건이 기술되어 있는가?

## LLD 필수 항목

> LLD는 필수 입력입니다. 아래 항목이 없으면 Spec 생성을 진행할 수 없습니다.

| ID | 항목 | blockOnMissing | 설명 |
|----|------|----------------|------|
| L-01 | 문제 정의 / 목표 | true | 이 LLD가 해결하려는 문제와 목표 |
| L-02 | FR (기능 요구사항) 목록 | true | 구현해야 할 기능 요구사항 (번호 있는 목록) |
| L-03 | 클래스 / 컴포넌트 설계 | true | 구현 클래스 목록, 책임, 관계 |
| L-04 | API 설계 | false | REST 엔드포인트, request/response 스펙 |
| L-05 | 시퀀스 다이어그램 | false | 핵심 흐름의 컴포넌트 간 상호작용 |
| L-06 | DB 스키마 | false | 테이블 구조, 필드, 인덱스, 관계 |
| L-07 | 설계 판단 / 트레이드오프 | false | 기술 선택 이유, 대안 검토 기록 |
| L-08 | 미해결 리스크 | false | 현재 불확실한 사항, 추후 결정 필요 항목 |
| L-09 | 검증 기준 | false | 각 FR이 올바르게 구현되었음을 확인하는 기준 |

### LLD 체크 항목

- [ ] **L-01** 문제 정의 및 구현 목표가 명시되어 있는가? _(blockOnMissing)_
- [ ] **L-02** 기능 요구사항(FR)이 번호 있는 목록으로 나열되어 있는가? _(blockOnMissing)_
- [ ] **L-03** 구현해야 할 클래스 또는 컴포넌트 목록과 각 책임이 기술되어 있는가? _(blockOnMissing)_
- [ ] **L-04** REST API 또는 인터페이스의 request/response 형식이 기술되어 있는가?
- [ ] **L-05** 핵심 흐름의 시퀀스 다이어그램 또는 호출 흐름이 존재하는가?
- [ ] **L-06** DB 테이블 또는 스키마 변경이 필요한 경우 스키마가 정의되어 있는가?
- [ ] **L-07** 설계 판단 사유, 대안 비교 기록이 존재하는가?
- [ ] **L-08** 현재 미해결 리스크나 불확실한 사항이 명시되어 있는가?
- [ ] **L-09** 각 FR에 대한 검증 기준(acceptance criteria)이 존재하는가?

## 조건부 항목

아래 항목은 입력에 해당 내용이 포함된 경우에만 확인합니다.

- [ ] **C-01** Kafka Consumer가 있는 경우: 이벤트 페이로드 구조가 정의되어 있는가?
- [ ] **C-02** Temporal Workflow/Activity가 있는 경우: 파라미터와 반환 타입이 명시되어 있는가?
- [ ] **C-03** Scheduler가 있는 경우: 스케줄 주기 및 실행 조건이 기술되어 있는가?
- [ ] **C-04** 외부 시스템 연동이 있는 경우: API 스펙 또는 인터페이스가 정의되어 있는가?

---

## 판정 기준

| 상태 | 조건 | orchestrator 동작 |
|------|------|------------------|
| **pass** | blockOnMissing 항목 전체 통과 | 다음 단계(design-analyzer) 진행 |
| **warn** | blockOnMissing=false 항목 1개 이상 누락 | 경고 로그 남기고 진행 |
| **fail** | blockOnMissing=true 항목 1개 이상 누락 | 사용자에게 보고, 진행 중단 |
| **override** | fail이지만 사용자가 오버라이드 지정 | 경고 남기고 진행 |

## 누락 보고 형식

blockOnMissing=true 항목이 누락된 경우 아래 형식으로 보고합니다:

```
[입력 품질 검증 실패]
누락 항목 (blockOnMissing):
  - L-01: 문제 정의/목표가 없습니다. LLD에 해결하려는 문제와 목표를 추가해주세요.
  - L-02: FR 목록이 없습니다. 기능 요구사항을 번호 목록으로 추가해주세요.

오버라이드하려면: --skip-input-check 옵션을 지정하세요.
```
