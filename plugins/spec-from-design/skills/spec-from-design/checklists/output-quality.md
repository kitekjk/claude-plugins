# 출력 품질 체크리스트 (output-quality)

> **사용 시점**: spec-reviewer(⑨)가 전체 Spec 세트 평가 시 적용
> **목적**: 생성된 Spec 파일의 품질을 100점 만점으로 평가
> **합격 기준**: 90점 이상 + auto-FAIL 조건 없음

---

## auto-FAIL 조건 (점수 무관)

아래 조건 중 하나라도 해당되면 점수에 관계없이 **즉시 FAIL** 처리합니다.

| ID | 조건 | 검출 방법 |
|----|------|----------|
| AF-01 | Spec 파일에 구현 코드 블록이 포함됨 (`\`\`\`java`, `\`\`\`kotlin` 등) | yaml dependsOn 블록 및 pseudocode 블록은 허용 |
| AF-02 | 1 Spec이 1 파일이 아님 (여러 UC가 한 파일에 존재) | 파일 내 `# SPEC-` 헤더 개수 확인 |
| AF-03 | Spec의 유형이 orchestrator가 지정한 유형과 불일치 | 기본 정보의 `유형:` 필드 비교 |
| AF-04 | model 또는 service 유형 Spec이 분해 없이 단독 생성됨 | 부모 usecase Spec 참조 없이 model/service 존재 |

### auto-FAIL 체크 항목

- [ ] **AF-01** 모든 Spec 파일에 구현 코드 블록(java/kotlin/python 등)이 없는가?
- [ ] **AF-02** 각 Spec 파일에 하나의 UC만 정의되어 있는가?
- [ ] **AF-03** 모든 Spec의 유형이 orchestrator 지정 유형과 일치하는가?
- [ ] **AF-04** model/service 유형이 있는 경우 분해 산출물로만 생성되었는가?

## 8카테고리 품질 평가 (100점 만점)

### Q1: 구조 완결성 (10점)

> 필수 섹션 존재 여부와 메타데이터 완전성을 평가합니다.

| 세부 항목 | 배점 | 체크 |
|----------|------|------|
| 기본 정보 섹션 존재 (유형, 도메인, 출처, 진입점 유형, 구현 클래스, 구현 상태) | 4 | [ ] |
| 개요, 수정 대상 파일, 기본 흐름, 검증 조건, 테스트 시나리오 섹션 존재 | 4 | [ ] |
| 파일명이 specPrefix 규칙을 따름 (`SPEC-{PREFIX}-{DOMAIN}-{number}-{name}`) | 2 | [ ] |

- [ ] **Q1** 구조 완결성: ___ / 10점

---

### Q2: 모호성 (15점)

> 자연어 표현의 명확성을 평가합니다. "적절히", "충분히" 같은 표현은 감점입니다.

| 세부 항목 | 배점 | 체크 |
|----------|------|------|
| "적절히", "충분히", "빠르게", "효율적으로" 등 모호한 표현 없음 | 5 | [ ] |
| 수치 기반 기술 (응답시간 < 200ms, 재시도 횟수 = 3회 등) | 5 | [ ] |
| 조건이 테스트 가능한 형태로 기술됨 (입력값 → 기대 결과) | 5 | [ ] |

- [ ] **Q2** 모호성: ___ / 15점

---

### Q3: 분기 커버리지 (20점)

> 정상 흐름 외 예외/대안 흐름이 충분히 커버되었는지 평가합니다.

| 세부 항목 | 배점 | 체크 |
|----------|------|------|
| 기본 흐름(정상 경로)이 단계별로 기술됨 | 5 | [ ] |
| 대안 흐름이 존재하고 각 분기 조건이 명시됨 | 5 | [ ] |
| LLD에 언급된 모든 예외/에러 케이스가 대안 흐름 또는 테스트 시나리오에 포함됨 | 5 | [ ] |
| 외부 시스템 장애, 타임아웃, 재시도 등 엣지 케이스 커버 | 5 | [ ] |

- [ ] **Q3** 분기 커버리지: ___ / 20점

### Q4: 참조 완결성 (10점)

> 다른 Spec, Policy, 모델에 대한 링크/참조가 유효한지 평가합니다.

| 세부 항목 | 배점 | 체크 |
|----------|------|------|
| 관련 정책 섹션의 Policy 참조가 실제 존재하는 Policy 파일을 가리킴 | 4 | [ ] |
| dependsOn에 나열된 Spec이 실제 생성된 Spec 파일과 일치함 | 3 | [ ] |
| 출처(LLD 참조)가 실제 LLD 섹션 또는 문서와 매칭됨 | 3 | [ ] |

- [ ] **Q4** 참조 완결성: ___ / 10점

| FR당 최소 1개 이상의 테스트 시나리오가 존재함 | 4 | [ ] |

### Q6: HLD/LLD 추적성 (10점)

> Spec의 각 내용이 HLD/LLD의 어느 부분에서 왔는지 추적 가능한지 평가합니다.

| 세부 항목 | 배점 | 체크 |
|----------|------|------|
| 기본 정보의 `출처:` 필드에 LLD 섹션/번호가 명시됨 | 4 | [ ] |
| 기본 흐름의 각 단계가 LLD의 FR과 대응됨 | 3 | [ ] |
| 관련 정책이 HLD KDD 또는 LLD 설계 판단에서 추출되었음 | 3 | [ ] |

- [ ] **Q6** HLD/LLD 추적성: ___ / 10점

> HLD/LLD에 없는 내용이 Spec에 추가되지 않았는지 평가합니다.

| 세부 항목 | 배점 | 체크 |
|----------|------|------|
| 기본 흐름에 LLD에 없는 단계가 추가되지 않음 | 4 | [ ] |
| 수정 대상 파일에 LLD에 없는 파일이 포함되지 않음 | 3 | [ ] |
| 대안 흐름/정책이 LLD 또는 HLD KDD에 근거가 있음 | 3 | [ ] |

- [ ] **Q7** YAGNI: ___ / 10점

### Q8: 스코프 적정성 (10점)

|----------|------|------|
| Spec이 하나의 구현 클래스(진입점)만 다룸 | 4 | [ ] |
| 수정 대상 파일이 ≤ 10개 (초과 시 분해 대상) | 3 | [ ] |
| FR이 ≤ 5개 (초과 시 분해 대상) | 3 | [ ] |

- [ ] **Q8** 스코프 적정성: ___ / 10점
