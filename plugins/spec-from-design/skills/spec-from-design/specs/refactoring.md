# Refactoring Spec 정의

동작 변경 없이 구조를 개선하는 변경에 사용합니다.
클래스 분리, 패턴 적용, 레거시 코드 제거, deprecation 등.
**구현 코드를 포함하지 않습니다**.

## 언제 사용하는가

- 기존 코드의 구조를 개선하되 동작은 동일하게 유지하는 경우
- 전략 패턴 적용, 인터페이스 추출, 클래스 분리 등
- 기존 엔드포인트 제거, 레거시 코드 정리

## 공통 섹션

기본 정보, 수정 대상 파일, dependsOn, 관련 정책, 관련 Spec, 관련 모델, 검증 조건, 테스트 시나리오

## refactoring 고유 섹션

| 섹션 | 설명 |
|------|------|
| 개요 | 리팩토링의 목적과 배경 |
| 변경 전 구조 | 현재 코드 구조를 자연어로 설명 |
| 변경 후 구조 | 목표 코드 구조를 자연어로 설명 |
| 변경 사유 | 왜 이 리팩토링이 필요한지 근거 |
| 영향 범위 | 리팩토링으로 영향받는 기존 기능/테스트 목록 |

## 핵심 제약

- **동작 변경 없음**: 리팩토링 전후 동일한 입력에 동일한 출력이 보장되어야 함
- 기존 테스트가 모두 통과해야 함
- 동작이 바뀌면 usecase Spec을 사용
