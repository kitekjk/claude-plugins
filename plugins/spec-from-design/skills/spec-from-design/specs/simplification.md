# simplification 유형 정의

## 목적

기존 구조의 복잡도를 줄이는 작업을 기술한다.
상태 통합, 불필요한 분기/배치 제거, 코드 경로 단일화 등을 포함하며,
refactoring과 달리 외부 동작(API, enum, DB 스키마) 변경을 수반할 수 있다.

## 생성 조건

- LLD에서 명시적으로 `simplification` 유형으로 분류한 항목만 생성
- LLD의 "단순화", "상태 통합", "배치 제거", "아키텍처 전환", "코드 경로 단일화" 명시 항목이 해당됨
- 외부 동작 변경(API 변경, enum 변경, DB 마이그레이션)을 수반하는 구조 개선인 경우

## refactoring과의 차이

| 구분 | refactoring | simplification |
|------|-------------|----------------|
| 외부 동작 | 불변 (기능 변경 없음) | 변경 가능 (API, enum, DB 스키마) |
| 범위 | 코드 구조 개선 | 아키텍처/상태 모델 수준 단순화 |
| 하위 호환성 | 유지 | 깨질 수 있음 (마이그레이션 필요) |
| 예시 | 클래스 분리, 중복 제거 | 상태 enum 통합, 배치 프로세스 제거, 동기→비동기 전환 |

## 필수 섹션

- 기본 정보 (유형, 도메인, 출처, 구현 상태)
- 개요 (단순화 목표 요약)
- Before 구조 (현재 구조, 복잡도 원인)
- After 구조 (변경 후 구조, 단순화 내용)
- 변경 사유 (단순화가 필요한 근거, 복잡도 비용)
- 영향 범위 (변경 파일, 연동 시스템, 하위 호환성)
- 테스트 시나리오 (Given-When-Then 형식, TC-ID 부여, 테스트 레벨 분류)

## 제약 사항

- LLD 명시 없이 임의 생성 금지
- 외부 동작이 변경되지 않으면 refactoring으로 분류
- 구현 코드 금지 (pseudocode 허용)
- 1 Spec = 1 파일
- 구현 상태 초기값: `not-started`
