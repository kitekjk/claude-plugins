# 비교 레벨 스펙

## 목적

독립 생성된 2개 구현체를 비교하여 Spec 결정성을 측정하는 5 Level 비교 체계를 정의한다.

## 핵심 원리

**차이나는 부분 = Spec이 모호해서 AI가 임의 판단한 부분**

차이가 클수록 Spec이 모호하고, 차이가 없을수록 Spec이 결정적이다.

## Level 정의

### L1: 테스트 상호 실행 (30점)

**측정**: regen1의 테스트를 regen2에서, regen2의 테스트를 regen1에서 실행

| 항목 | 배점 | 기준 |
|------|------|------|
| 양방향 컴파일 성공 | 10점 | 양쪽 모두 성공 = 10점 |
| 양방향 테스트 통과율 | 20점 | 통과율 평균 × 20점 |

**컴파일 실패 시사점**: 클래스명/메서드명이 다름 → naming-guide 모호
**테스트 실패 시사점**: 비즈니스 판정이 다름 → policies 규칙 모호

### L2: API 계약 일치 (25점)

**측정**: 동일 요청 → 응답 구조 비교

| 항목 | 배점 | 기준 |
|------|------|------|
| 엔드포인트 구조 일치율 | 25점 | 일치율 × 25점 |

**비교 대상**: HTTP 상태 코드, 응답 JSON 필드명 + 타입, 에러 응답 형식
**제외 대상**: id, createdAt, updatedAt, token 등 동적 값

**필드 차이 시사점**: API Spec에 응답 필드 미명시
**상태 코드 차이 시사점**: 에러 처리 미명시

### L3: 비즈니스 시나리오 일치 (25점)

**측정**: 업무 흐름 실행 후 결과 상태 비교

| 항목 | 배점 | 기준 |
|------|------|------|
| 시나리오 판정 일치율 | 25점 | 일치율 × 25점 |

**시나리오 추출 소스**: Use Case Spec, policies, 테스트 시나리오
**비교 대상**: 비즈니스 판정 (상태, 평가, 금액) — 동적 값 제외

**판정 차이 시사점**: 비즈니스 규칙 모호
**금액 차이 시사점**: 계산 규칙 모호 (반올림, 절사 등)

### L4: 코드 구조 일치 (10점)

**측정**: 구조적 패턴 일치 여부 (코드가 같을 필요는 없음)

| 항목 | 배점 | 기준 |
|------|------|------|
| 아키텍처 패턴 일치 | 10점 | 일치 항목 수 비례 |

**비교 대상**:
- 모듈/레이어 분리
- 패키지 구조 (도메인별 분리)
- 아키텍처 패턴 적용 여부 (architecture-rules에 정의된 패턴들)

**패턴 차이 시사점**: architecture-rules 규칙 부족

### L5: 네이밍 일치 (10점)

**측정**: 클래스명, API 경로, Enum 이름 일치 여부

| 항목 | 배점 | 기준 |
|------|------|------|
| 클래스명/API/Enum 일치율 | 10점 | 일치율 × 10점 |

**비교 대상**:
- 핵심 클래스명 (Entity, VO, Repository, Controller)
- API 경로
- Enum 이름 + 값

**차이 시사점**: naming-guide 규칙 부족

## 등급 체계

| 점수 | 등급 | 의미 | 후속 조치 |
|------|------|------|-----------|
| 95~100 | A | 우수 | 프로덕션 사용 가능 |
| 85~94 | B | 양호 | 소규모 보강 후 사용 가능 |
| 70~84 | C | 보통 | Spec 보강 필요 |
| 70 미만 | D | 부족 | 대폭 보강 필요 |
